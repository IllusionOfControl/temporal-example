# Референсный проект Temporal (Python SDK)

Данный репозиторий содержит примеры реализации отказоустойчивых распределенных процессов с использованием **Temporal** и современного стека Python (`temporalio`, `asyncio`, `Pydantic`, `SQLAlchemy 2.0`). Цель проекта — демонстрация микросервисных паттернов, изоляции вычислительных ресурсов (Task Routing) и управления состоянием длительных транзакций.

## Развертывание

Для локального запуска инфраструктуры (Temporal Server, PostgreSQL, Wiremock) и пула воркеров используется Docker Compose.

### Предварительная настройка
Создайте файл переменных окружения на основе примера:
```shell
cp .env.example .env
```

### Управление средой
Сборка образов (используется `uv` для сверхбыстрой установки зависимостей):
```shell
make build
```
Запуск всех сервисов:
```shell
make up
```
Остановка среды:
```shell
make down
```

После запуска Web UI Temporal будет доступен по адресу: **http://localhost:8233**

## Описание Workflow-сценариев

### 1. Базовый процесс (`workflows/greeting.py`)
Демонстрирует минимально необходимый сетап для запуска Temporal Workflow.
* **Технологический стек:** `@workflow.defn`, `@activity.defn`.
* **Ключевые особенности:** 
    * Изоляция импортов через `workflow.unsafe.imports_passed_through()`.
    * Обязательное указание `start_to_close_timeout` для защиты от зависания активностей.

### 2. Интеграция с API и БД (`workflows/user_onboarding.py`)
Процесс регистрации с обращением к внешнему сервису проверки документов (KYC).
* **Технологический стек:** `RetryPolicy`, SQLAlchemy (async), `httpx`.
* **Ключевые особенности:** 
    * **Идемпотентность:** Использование `ON CONFLICT DO NOTHING` в БД. Гарантирует безопасность при автоматических ретраях Temporal.
    * **Политики повтора:** Настройка `RetryPolicy` для HTTP-запросов (Wiremock) на случай сетевых сбоев.
    * **Внедрение зависимостей (DI):** Передача пула соединений БД и API-клиентов в классы активностей при инициализации воркера.

### 3. Human-in-the-loop и Параллелизм (`workflows/order_processing.py`)
Сложный пайплайн обработки заказа электронной коммерции.
* **Технологический стек:** Signals, Queries, `asyncio.gather`, `wait_condition`.
* **Ключевые особенности:**
    * **Queries:** Использование `@workflow.query` для получения текущего статуса заказа извне без прерывания процесса.
    * **Signals:** Приостановка выполнения (`await workflow.wait_condition`) до получения внешнего асинхронного события (`@workflow.signal` — подтверждение менеджером). Процесс может спать месяцами, не потребляя CPU.
    * **Параллельное выполнение:** Одновременный запуск независимых активностей (генерация чека и отправка email) через `asyncio.gather`.

### 4. Маршрутизация задач и ML-пайплайн (`workflows/review_ml.py`)
Пайплайн обработки данных с разделением нагрузки по типам оборудования.
* **Технологический стек:** Task Queues, Heartbeats, Pydantic Data Converter.
* **Ключевые особенности:**
    * **Task Routing:** Направление легких задач в `CPU_TASK_QUEUE`, тяжелых ML-моделей в `GPU_TASK_QUEUE`, а работы с БД — в `IO_TASK_QUEUE`.
    * **Heartbeats:** Использование `heartbeat_timeout` для длительных вычислений (суммаризация текста), чтобы Temporal понимал, что воркер жив, даже если задача выполняется долго.
    * **Строгая типизация:** Использование `pydantic_data_converter` для автоматической сериализации/десериализации Pydantic-моделей между клиентом и воркерами.

## Структура и стандарты разработки

Проект строго следует принципу разделения ответственности (Separation of Concerns) и архитектуре Temporal:

* **`src/app/workflows/`**: Содержит **только** детерминированную бизнес-логику оркестрации. Запрещены прямые сетевые вызовы, работа с БД, генерация случайных чисел и работа со временем (кроме `workflow.time()`).
* **`src/app/activities/`**: Недетерминированный код. Взаимодействие с внешним миром (БД, API, файловая система, ML-модели).
* **`src/app/workers/`**: Точки входа для процессов-исполнителей. Разделены по профилю нагрузки (`main`, `io`, `cpu`, `gpu`).
* **`src/app/infrastructure/`**: Настройка подключений к БД, HTTP-клиенты и загрузчики моделей.
* **`include/`**: Внешние ресурсы (SQL-миграции, моки Wiremock).

### Запуск тестовых сценариев
Для инициации процессов предусмотрены клиентские скрипты в корне проекта. Запускать их следует при поднятом Docker Compose:
```shell
# Запуск простого приветствия
python run-hello-workflow.py

# Запуск онбординга пользователя
python run-user-onboard-workflow.py

# Запуск обработки заказа (демонстрация сигналов и запросов)
python run-order-processing-workflow.py

# Запуск ML-анализа отзывов
python run-review-workflow.py
```

### Требования к разработке
1. **Детерминированность Workflows**: Любая логика, которая может дать разный результат при повторном запуске, должна быть вынесена в Activity.
2. **Таймауты**: Каждая активность обязана иметь явно заданный `start_to_close_timeout`.
3. **Идемпотентность Activities**: Код активностей должен быть написан так, чтобы его многократное выполнение (из-за ретраев) не приводило к дублированию данных или некорректным состояниям системы.

## Тестирование

Проект включает настроенную инфраструктуру для автоматизированного тестирования с использованием `pytest` и встроенного тестового окружения Temporal (`temporalio.testing`).

### Ключевые особенности тестирования Temporal
1. **Time Skipping (Промотка времени):** Тестовый сервер Temporal (`WorkflowEnvironment.start_time_skipping()`) позволяет мгновенно "проматывать" таймауты, `asyncio.sleep` и длительные ожидания сигналов. Воркфлоу, рассчитанный на месяцы работы, в тестах выполняется за миллисекунды.
2. **Изоляция активностей (Unit-тесты):** Бизнес-логика активностей тестируется изолированно с помощью `ActivityEnvironment` без необходимости поднимать сервер оркестрации.
3. **Мокирование (Mocking):** При интеграционном тестировании Workflows реальные активности (походы в БД, вызовы API) подменяются фейковыми функциями-заглушками. Связывание происходит по имени активности: `@activity.defn(name="real_activity_name")`.
4. **Точная имитация архитектуры (Task Routing):** В тестах поднимается несколько локальных in-memory воркеров, каждый из которых слушает свою очередь (`MAIN_TASK_QUEUE`, `IO_TASK_QUEUE`, `CPU_TASK_QUEUE`). Это гарантирует, что маршрутизация задач в тестах работает точно так же, как в production-среде.

### Запуск тестов
Для запуска тестов убедитесь, что установлены зависимости для разработки (`pytest`, `pytest-asyncio`), и выполните команду из корня проекта, указав путь к исходному коду:

```shell
make tests
